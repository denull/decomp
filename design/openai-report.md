# Comparative Review of Modern UI Component Libraries

## 98.css  
A retro-inspired CSS theme library that makes plain HTML look like Windows 98. Its docs list standard form and window elements (Buttons, Checkbox, OptionButton, GroupBox, TextBox, Slider, Dropdown, Title Bar, TreeView, Tabs, TableView, Progress Indicator, Field borders, etc.)【4†L2-L9】【4†L15-L19】. 98.css is purely CSS (no JavaScript) and uses semantic HTML, so it works with any framework【4†L30-L33】【4†L50-L52】.  **Strengths:**  A unique “Windows 98” aesthetic and very simple integration.  **Weaknesses:** Single fixed theme (no alternate skins or easy color changes) and minimal interactivity beyond CSS; heavy reliance on its specific style means you must override a lot of CSS for a different look.

## shadcn (shadcn-svelte)  
A comprehensive, Tailwind-based component system (originally for React) that’s been ported to Svelte. It advertises itself as “a set of beautifully designed components that you can customize, extend, and build on”【8†L19-L22】. Indeed, its docs list dozens of components (Accordion, Alert, Avatar, Badge, Button, Card, Carousel, Checkbox, Dialog, Dropdown Menu, Input, Menu, Navigation, Popover, Tabs, Tooltip, Typography, etc.)【12†L136-L144】. shadcn-svelte uses Tailwind CSS under the hood and adopts a CSS-variable theming scheme – e.g. color tokens like `--primary` and `--primary-foreground` are used throughout【11†L136-L143】. This makes it easy to override colors via CSS variables or your Tailwind config.  **Strengths:** Very large component set with modern design and built-in dark mode examples. Default styles are nice, and because it’s Tailwind-based you can theme it via CSS variables or Tailwind config.  **Weaknesses:** Tightly coupled with Tailwind (less useful if you don’t use Tailwind), and the number of components means a steeper learning curve. It also requires a build step (using Tailwind) and isn’t “headless” – it comes with opinions on styling, though mostly via utility classes.

## Base UI (MUI Base)  
An unstyled “headless” component library from the MUI team (creators of Material-UI and Joy UI). Its docs emphasize “unstyled UI components for building accessible user interfaces”【16†L6-L14】. The component list is broad (Accordion, AlertDialog, Autocomplete, Avatar, Button, Checkbox, Dialog, Menu, Popover, Switch, Tabs, Tooltip, etc.)【17†L31-L39】【17†L53-L60】. Importantly, Base UI provides *no* styling by default – you assemble parts and apply your own CSS or CSS-in-JS as shown in their quickstart (e.g. examples use Tailwind or CSS Modules to style a Popover)【20†L152-L155】. It’s also fully tree-shakable.  **Strengths:** Maximum flexibility and accessibility; components are designed without visual opinions so you can integrate any design system.  **Weaknesses:** Out-of-the-box there is no styling or theme at all – you must supply it yourself. It’s React-only (with Svelte or others outside its scope), and being a low-level primitive set, using it can feel verbose. 

## DaisyUI  
A Tailwind CSS plugin that adds pre-styled components as utility classes. It is **not** a Svelte library but a collection of CSS classes and themes for Tailwind. DaisyUI lists “65 components” (accordions, alerts, badges, breadcrumbs, buttons, cards, carousels, chat bubbles, collapses, dropdowns, modals, navbars, tables, tabs, toast, etc.)【24†L124-L127】. It also provides 35 built-in color themes (light, dark, retro, fantasy, etc.), making theme-switching easy via data attributes or CSS classes【27†L201-L207】.  **Strengths:** Very easy to integrate into any Tailwind project; numerous ready-to-use styles and built-in themes (including dark mode). **Weaknesses:** It’s only CSS classes (no Svelte components), so behavior must be implemented separately. And theming is limited to its predefined palettes (though you can customize via Tailwind config if needed).

## Radix UI  
A framework-agnostic library of low-level, unstyled “primitives” focused on accessibility. It provides components like Accordion, AlertDialog, Avatar, Checkbox, Dialog, Dropdown Menu, HoverCard, Menubar, NavigationMenu, Popover, Progress, RadioGroup, ScrollArea, Select, Slider, Switch, Tabs, Toast, Toggle, Tooltip, etc.【32†L26-L34】.  All Radix components are **unstyled by default**, giving you “complete control over the look and feel”【32†L80-L84】. This makes them flexible for any theming system, but you must supply CSS or utility classes.  **Strengths:** Robust accessibility and highly composable primitives (also well-tested). **Weaknesses:** No styling provided, so it requires extra work to style or theme; only for React. Its focus is on complex widgets (menus, popovers, etc.) rather than simple presentational elements.

## HeroUI (v3)  
A polished React component library (formerly NextUI) with a modern design. The site calls it a “modern UI library for web and mobile, built to help you move fast”【34†L19-L23】. It includes dozens of styled components (Accordion, Alert, Avatar, Badge, Button, Card, Calendar, Checkbox, DatePicker, Dialog, Drawer, Input, Menu, Pagination, Popover, Slider, Tabs, Toast, Tooltip, etc.) as shown in its docs【37†L15-L23】. HeroUI is opinionated in style (“beautiful by default”) but supports theming via CSS variables and BEM-style classes【40†L52-L55】 – for example, you can tweak colors or enable dark mode through its theme API.  **Strengths:** Great default look and a visual theme builder; plenty of features and customization options. **Weaknesses:** React-only; theming requires understanding its CSS variables/BEM system; less flexible than headless libraries (it has its own style rules).

## Chakra UI  
A popular React component library known for its simplicity and theming. Chakra offers a large set of accessible components (Button, Form Controls, Alert, Drawer, Modal, Tabs, Menu, etc. – see overview【42†L113-L121】) and a powerful theme system. Theming is based on design tokens and a `ChakraProvider` context; you configure colors, typography, spacings in a JavaScript theme (based on the Panda CSS engine)【44†L45-L53】. Chakra’s components use CSS variables behind the scenes for colors and modes, so you can dynamically switch light/dark or customize tokens. **Strengths:** Extensive component set, dark mode out-of-the-box, intuitive API, mobile-first design. **Weaknesses:** Tied to React and its theming approach; adding custom theming beyond the provided tokens can be complex.

## Headless UI  
A minimalistic toolkit from the creators of Tailwind. It provides fundamental interactive components (Menu (dropdown), Listbox, Combobox, Switch, Disclosure (accordion), Dialog (modal), Popover, RadioGroup, Tabs, Transition, etc.)【49†L11-L19】. All components are completely **un-styled** and only handle behavior and accessibility – you apply your own Tailwind (or other) classes.  **Strengths:** Seamless Tailwind integration and solid accessibility; great for when you want total control of styling but need keyboard support. **Weaknesses:** No pre-made visuals or theme – purely logic/structure. Only available for React and Vue (no Svelte). 

## Mantine  
A React component library with 100+ components and hooks (buttons, inputs, alerts, modals, datepickers, charts, etc.). Mantine has built-in theming: “all components are built with the Mantine theme, change colors, fonts, shadows…” and it fully supports dark/light modes by default【51†L17-L25】. You configure a theme object (with color schemes and design tokens) and wrap your app in a provider. Mantine’s components are also responsive and fairly lightweight.  **Strengths:** Large, well-documented, with ready dark/light mode. **Weaknesses:** React only; styling is not Tailwind-based (uses Emotion/CSS-in-JS), so theming requires using its provider and tokens.

## Quasar Framework (Vue)  
A comprehensive Vue (and Svelte via wrappers) framework that includes a huge component library (over 80 UI components from Avatars and Buttons to Tables, Charts, and Maps). For example, Quasar lists: Avatar, Badge, Banner, Bar, Breadcrumbs, Button (and Button Group/Toggle), Card, Carousel, Chip, Checkbox, Input, Dialog, Drawer, Fab, Footer, File Picker, Form, Field, List, Menu, Pagination, Popover, Progress (circular/linear), Radio, Slider, Switch, Tabs, Table, Timeline, Toolbar, Tooltip, Tree, etc. (see Quasar Docs【55†L61-L69】【55†L73-L80】). Quasar has a built-in **Theme Builder** (and dark mode support)【55†L39-L42】. You can pick primary color, define custom variables via SCSS, and toggle dark/light.  **Strengths:** Extremely complete and integrated (also PWA/mobile-ready). **Weaknesses:** Heavier framework (Vue-centric); theming mostly via SCSS variables and config, which is powerful but more complex. Not natively Svelte, though some components can be used.

## Tailwind CSS + Plus (Tailwind Plus)  
Not a component library per se, but an official collection of **templates and snippets** built with Tailwind. It includes 500+ hand-crafted UI component examples (built for React/Next) in the Tailwind UI style【56†L0-L4】. There are buttons, forms, navbars, cards, etc., but they’re delivered as code examples or blocks to copy – no interactive Svelte components.  **Use case:** inspiration or prototyping with Tailwind.  **Weaknesses:** No actual components to install or theme dynamically – it’s static examples.

## Flowbite Svelte  
A Svelte-tailored library based on the Flowbite (Tailwind) design system. The site emphasizes that “Flowbite Svelte is an open-source UI component library built with Svelte components and Tailwind CSS”【58†L158-L160】. It provides 60+ components (dropdowns, navbars, modals, inputs, forms, cards, etc.) and includes hooks for dark mode【58†L69-L72】【61†L7-L10】. According to its docs, components are interactive via Svelte, styled with Tailwind classes, and follow the Flowbite design tokens【61†L18-L21】.  **Strengths:** Combines Svelte reactivity with Tailwind styling, and comes with a built-in dark mode toggle. **Weaknesses:** Less flexibility than headless libraries (relies on Tailwind classes), and components may not fit non-Tailwind projects.

## Svelte Material UI (SMUI)  
A Svelte wrapper around Material Design Components (MDC). It offers many Material UI widgets: Buttons, Text Fields, FABs, Cards, Dialogs, Drawers, Menus, Lists, Data Table, Snackbar, Tabs, etc. (see Quick Guide list【63†L23-L32】). All components are fully “Material Themed” and follow Google’s spec. SMUI uses Sass variables (from MDC) for theming: you can override color, shape, density, etc. via SCSS【63†L81-L85】. **Strengths:** Authentic Material design look & feel, good accessibility, Svelte-specific. **Weaknesses:** Larger bundle (material components), theming only via SCSS (no dynamic theme switching at runtime). 

## Skeleton UI  
A Svelte + Tailwind design system. It bills itself as “the adaptive design system for Tailwind CSS”【65†L178-L182】. Skeleton provides “turnkey” Svelte components for forms, navigation, dialogs, etc., and supports multiple style themes (notably a built-in “Catppuccin” theme set, among others). Components listed include DatePicker, Dialog, Listbox, Menu, Pagination, Popover, Portal, Progress (circular/linear), Rating, Slider, Steps, Switch, Tabs, Toast, Tooltip, TreeView, etc.【65†L51-L59】【65†L60-L68】. They are styled using Tailwind utility classes. **Strengths:** Svelte-centric with lots of components and unique themes; styling via Tailwind and CSS variables for colors. **Weaknesses:** Relatively new and less battle-tested; theming is mostly via CSS classes and custom properties, so not as flexible as a full JS theme system.

## Bits UI  
A “headless” component library for Svelte (framework-agnostic primitives). Its docs say “Flexible, unstyled, accessible primitives”【70†L10-L13】. Available components include Accordion, Avatar, Calendar, Checkbox, Combobox, ContextMenu, DatePicker, Dialog, DropdownMenu, Menubar, Pagination, Popover, Progress, RadioGroup, Select, Slider, Switch, Tabs, Tooltip, etc.【72†L10-L18】. By default *nothing is styled* – the user “bring the styles using class props or data-attributes”【72†L73-L78】.  **Strengths:** Highly flexible and accessible, suitable for building a custom design system; works with any styling approach. **Weaknesses:** No built-in visuals (you must style every component) and not as “turnkey” as opinionated libraries.

## Melt UI  
Another Svelte headless library of “builders” (similar in concept to Radix/BitsUI). It provides the same kinds of components (Accordion, Avatar, Calendar, Checkbox, Dialog, DropdownMenu, Menubar, Pagination, Popover, Select, Slider, Switch, Tabs, Toast, Tooltip, Tree, etc.) – see its docs’ list【76†L22-L30】. Melt emphasizes accessibility and “zero predefined styles”【76†L135-L140】. Like BitsUI, you supply all styling (via CSS, Tailwind, etc.), and its builders are composable stores/actions. **Strengths:** Modern Svelte design, focus on ARIA and flexibility. **Weaknesses:** Completely unstyled and code-centric, so more setup; Svelte-specific only.

## Origin UI Svelte  
A Svelte collection of copy-paste UI components inspired by Origin UI (now MUI Base). The site tagline is “Beautiful UI components built with Tailwind CSS and Svelte”【78†L7-L13】. It features dozens of themed examples (Accordions, Alerts, Avatars, Badges, Banners, Breadcrumbs, Buttons, Checkboxes, Dialogs, Dropdowns, Inputs, Navbars, Notifications, Radios, Selects, Tables, Tabs, Tooltips, etc.) grouped into categories on the demo site【78†L19-L51】. These are static components (HTML/CSS/Tailwind) you paste in. **Strengths:** Many ready designs and color variants; uses Tailwind for easy theming via classes. **Weaknesses:** Not componentized – it’s more a set of examples/templates, so less reusable logic and no interactive behavior out-of-the-box.

## Magic UI  
A React-focused library of **animated UI components** and landing-page effects. Its site boasts “150+ free open-source animated components and effects built with React, Tailwind CSS, and Motion”【80†L25-L28】. It includes fanciful widgets (text animations, background effects, buttons, spinners, etc.) alongside some standard UI patterns. It even has a “Theme Toggler” component. **Use case:** stylized landing pages and marketing sites. **Weaknesses:** React-only, very heavy on animations, and not suited for building standard app UIs or non-animated form UIs.

## Ant Design  
A mature React UI framework by Alibaba, with a very comprehensive set of components (Buttons, Forms, Layouts, Navigation, Data Display, Feedback, etc. – see their overview【83†L268-L277】【83†L352-L359】 for the full list). AntD uses a *design-token* theming system: since v5 it relies on CSS-in-JS with a `ConfigProvider` to dynamically supply tokens for colors, border radii, etc. The docs note you can “customize our design tokens” (primary color, border radius, etc.) and even switch themes at runtime【84†L239-L247】. **Strengths:** Very complete and consistent enterprise-ready components; flexible theming with multiple presets (light/dark/compact) and custom tokens. **Weaknesses:** Large bundle size; React-only; theming requires use of their token system or altering Less/CSS (and legacy themes are less straightforward). 

## AgnosticUI  
An unconventional approach: components are copied directly into your project (via a CLI) so they “live in your code”【86†L10-L14】【86†L54-L61】. It supports multiple frameworks (React, Vue, Svelte, etc.) by generating appropriate files. The philosophy is “AI-native” with full code transparency. Basic components (Button, Card, Input, etc.) come unstyled so you can theme them as you wish.  **Strengths:** Full control and no hidden implementation – themes and styles are up to you. **Weaknesses:** It’s essentially code scaffolding, not a themable library out-of-the-box; you must manage the styling entirely yourself.

## Open Props  
Not a component library but a collection of **CSS custom properties (design tokens)**. It provides a huge set of ready-made CSS variables for colors, spacing, typography, etc. Open Props includes normalized themes for light and dark modes (and even “dim” and “purple” color schemes) via CSS variables【89†L1211-L1220】【90†L53-L58】. In practice, you import Open Props and use their variables (e.g. `var(--surface-1)`, `var(--text-1)`) in your own CSS or components. **Use case:** building your own themable design system with minimal effort, since all base tokens are defined. 

## Open Props UI  
A lightweight, copy-and-paste UI library built on Open Props. It advertises “The UI library of the future… Components you can copy and paste to your project”【91†L25-L33】. All components (Accordion, Alert, Avatar, Badge, Button, Card, Checkbox, Dialog, Progress, Radio, Switch, Table, TextField, etc.) are pure HTML/CSS, leveraging modern CSS for interactivity. Styling is entirely via Open Props variables and utility classes – it’s **CSS-only** with no JS behavior【91†L25-L33】. **Strengths:** Complete control and minimal footprint; you tweak CSS directly. **Weaknesses:** Very basic (few interactive features); all logic must be added manually if needed; still early-stage with a limited component set.

## Other Notable Libraries  
A few other projects are worth mentioning. **Shoelace** is a framework-agnostic *web component* library (not Svelte-specific) that offers a wide range of components (buttons, dropdowns, tooltips, etc.) with built-in theming via CSS variables【95†L1-L3】. **FAST (Microsoft)** is a toolkit for building Web Components with design tokens; it underpins Fluent UI and emphasizes flexibility and modern standards【98†L11-L15】. **Carbon Components Svelte** (IBM’s design system) provides dynamic theming: it supports several pre-defined themes (white, g10, g80, g90, g100) using CSS variables【99†L1-L3】. Other Svelte-friendly kits include **Sveltestrap** (Bootstrap components for Svelte), **YeSvelte** (Bootstrap-based), **Svelte UX** (large interactive component set), **Smelte** (Material+Tailwind), **M3 Svelte** (Material 3), **Quaff** (Material 3 design for Svelte), and **retroui-svelte** (retro-themed). Generally, these either wrap existing CSS frameworks (Bootstrap/Material) or provide specialized UI patterns; theming is done via the underlying framework’s variables or CSS.

### Theming Patterns Across Libraries  
Different libraries adopt different theming approaches: 

- **CSS Variables / Design Tokens:** Several libraries expose CSS custom properties for colors, so you can swap themes by overriding variables. *Examples:* Shadcn-svelte uses CSS variables for its color scheme【11†L136-L143】; HeroUI provides CSS-vars + BEM theming【40†L52-L55】; Carbon Svelte uses CSS variable tokens for theme variants【99†L1-L3】. Open Props is entirely built on CSS variables, encouraging aliasing strategies for light/dark themes.

- **Utility-Class Themes (Tailwind):** Tailwind-based libraries rely on utility classes and pre-built palettes. *Examples:* DaisyUI has 35 ready themes (light, dark, etc.) selectable via a `data-theme` attribute【27†L201-L207】. Skeleton UI and Flowbite Svelte also use Tailwind classes and often include dark mode classes. These allow quick theming by choosing a predefined palette or configuring Tailwind’s theme settings.

- **Context/Token-Based Themes:** Some libraries use a JavaScript theme object. *Examples:* Chakra UI and Mantine employ a context provider where you define color/token scales in JS. Ant Design uses a `ConfigProvider` with design tokens for colors, spacing, enabling runtime switching of light/dark themes【84†L239-L247】. These systems let you generate multiple themes or switch on the fly.

- **Unstyled / Bring-Your-Own:** Many “headless” libraries (Radix, Base UI, HeadlessUI, BitsUI, MeltUI) ship *unstyled*. They expect you to supply all CSS or utility classes, meaning the library itself imposes no theme. For instance, Radix’s docs explicitly say components come without styles and you style them yourself【32†L80-L84】; BitsUI notes “you bring the styles using standard class props or data-* attributes”【72†L73-L78】.

- **Fixed / Retro Themes:** A few have a single fixed aesthetic. 98.css is an example – it only offers the Windows-98 look with no alternate skins【4†L30-L33】. Likewise, static component templates (like Origin UI Svelte examples or TailwindPlus templates) have hard-coded styles unless manually overridden. 

In summary, theming ranges from CSS-variable-driven systems and JS-theme providers to simple Tailwind palettes or “no theme” headless approaches. Modern libraries tend to favor CSS variables or design tokens for flexibility, whereas utility libraries leverage Tailwind’s theming. The right pattern depends on whether you need full control (headless/CSS-vars) or convenience (preset themes with Tailwind or CSS-in-JS).
